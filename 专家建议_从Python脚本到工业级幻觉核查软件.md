# 从 Python 脚本到工业级可长期发展的幻觉核查软件：全流程专家级落地指南

本指南完全围绕「长期可持续发展」的核心目标设计，摒弃野路子一次性打包的短视方案，从根基规范化、核心竞争力构建、产品化交付、生态化扩展、合规商业化五大核心维度，拆解全流程的专业细节、落地标准、避坑要点和长期价值设计，确保你的软件不仅能当下可用，更能在未来 3-5 年持续迭代、形成壁垒、甚至实现商业化。

## 前置核心认知：野路子脚本 vs 专业级软件的本质区别

你当前的脚本，本质是「一次性可运行代码片段」，核心问题是无规范、无边界、无沉淀、不可控，这也是它只能藏在本地隐藏文件夹、只有 AI 知道怎么启动的核心原因。而能长期发展的专业软件，必须满足 5 个核心工业级标准：

- 可维护性：任何人接手代码都能快速看懂、修改、迭代，不会出现 "换个人就废了" 的情况
- 可确定性：运行结果可复现、错误可追溯、行为可预测，不会出现 "在我这能跑在你那崩了" 的玄学问题
- 可扩展性：新增功能、适配新场景、对接新模型时，无需重构核心代码，可渐进式升级
- 合规可控性：数据、知识产权、隐私、监管全链路合规，无法律风险
- 用户价值闭环：从打开软件到完成核查、导出结果，全流程开箱即用，无需任何代码基础

## 第一阶段：根基固化 —— 项目规范化与资产全生命周期管理

这一阶段的核心目标，是把你散落在本地隐藏文件夹的脚本，变成可沉淀、可追溯、可管理的软件核心资产，彻底解决 "只有 AI 知道在哪、改几次就乱了" 的问题，也是软件长期发展的地基。

### 1.1 工业级 Python 项目目录结构重构（强制标准）

摒弃单文件 main.py 的野路子写法，采用 Python 社区公认的 src-layout 源码分离架构（符合 PEP 621 标准），从根源上解决导入错误、路径混乱、打包失败、隐藏目录乱存数据的问题。

#### 标准目录结构及每一层的长期设计意义

```
hallucination_checker/
├── pyproject.toml
├── README.md
├── CHANGELOG.md
├── LICENSE
├── src/
│   └── hallucination_checker/
│       ├── __init__.py
│       ├── __main__.py
│       ├── engine/
│       ├── io/
│       ├── cli/
│       └── gui/
├── tests/
├── config/
└── docs/
```

#### 关键设计细节

**彻底解决隐藏目录乱存数据的问题：** 摒弃自定义隐藏文件夹，严格遵循跨平台的 XDG 基础目录规范：

| 操作系统 | 配置文件路径 | 用户数据路径 | 缓存路径 |
|----------|-------------|-------------|---------|
| Windows | %LOCALAPPDATA%/HallucinationChecker/config | %APPDATA%/HallucinationChecker/data | %TEMP%/HallucinationChecker/cache |
| macOS | ~/Library/Preferences/HallucinationChecker | ~/Library/Application Support/HallucinationChecker | ~/Library/Caches/HallucinationChecker |
| Linux | ~/.config/HallucinationChecker | ~/.local/share/HallucinationChecker | ~/.cache/HallucinationChecker |

**核心逻辑与 UI/IO 完全解耦：** 核心核查引擎完全独立，哪怕后续你把 GUI 从 Qt 换成 Web、换成移动端，核心代码一行都不用改。

### 1.2 版本控制与迭代规范

- 强制使用 Git 进行全链路版本管理
- 仓库托管：国内用 Gitee，国际用 GitHub
- 分支管理策略：Git Flow（main/develop/feature/xxx/hotfix/xxx）
- 语义化版本号（SemVer 2.0）：主版本号.次版本号.修订号
- 强制 CHANGELOG.md 更新日志规范

### 1.3 依赖管理与环境隔离

- 使用 Poetry 或 PDM 作为依赖管理工具
- 依赖分层设计：核心依赖 / 可选依赖 / 开发依赖 / 锁定依赖
- 提供 Dockerfile 实现环境一致性

### 1.4 代码质量与可维护性管控

- 强制类型注解（PEP 484）+ mypy 静态检查
- Ruff 代码格式化与 lint
- pytest 单元测试，核心引擎覆盖率 ≥90%
- 异常处理标准化（摒弃裸 except）
- logging 模块分级日志系统（DEBUG/INFO/WARNING/ERROR/CRITICAL）

## 第二阶段：核心壁垒构建 —— 幻觉核查引擎的专业化与工程化重构

### 2.1 工业级幻觉核查引擎的标准架构

全链路闭环：输入预处理 → 事实单元拆分 → 多源交叉校验 → 幻觉分级量化 → 证据链生成 → 结果输出

#### 各模块设计

| 模块 | 核心功能 | 专业实现细节 |
|------|---------|-------------|
| 输入预处理模块 | 文本清洗、格式解析、分块处理 | 支持 txt/md/docx/pdf/html 统一转纯文本 |
| 声明提取模块 | 拆分为最小事实单元 | 句法分析 + LLM 指令微调，分配唯一 ID |
| 多源事实校验模块 | 多维度交叉验证 | 权威知识库 + 联网搜索 + 多模型投票 + 规则引擎 |
| 幻觉分级与量化 | 标准化分级和打分 | 3 级（严重/中度/轻微）+ 0-100 分可信度 |
| 证据链生成模块 | 可溯源证据链 | 权威来源原文 + 链接 + 发布时间 + 信源等级 |
| 结果输出模块 | 统一格式化结果 | 原文高亮 + JSON/CSV + PDF/Word/Excel 报告 |

### 2.2 核心工程化能力补齐

- 适配层隔离设计：统一接口封装外部依赖
- 容错与重试机制：超时 + 重试 + 降级策略
- 性能优化：异步 IO + 并行处理 + 缓存机制
- 内存与资源管控：大文件内存限制 + 临时文件清理

## 第三阶段：产品化交付 —— 从代码到开箱即用的跨平台软件

### 3.1 GUI 架构选型与开发

**首选方案：Qt 框架（PySide6/PyQt6）**
- 跨平台（Windows/macOS/Linux）
- MVVM 架构，界面与业务逻辑分离
- 完整用户交互闭环（启动页/主界面/配置/结果展示/历史记录）

**备选方案：Electron + FastAPI 后端**（为 SaaS 化铺路）

### 3.2 工业级跨平台打包与分发

| 平台 | 打包方案 | 合规处理 | 分发形式 |
|------|---------|---------|---------|
| Windows | PyInstaller + Inno Setup | EV 代码签名证书 | exe 安装包 / Microsoft Store / winget |
| macOS | PyInstaller (.app) + create-dmg | 开发者 ID 签名 + 苹果公证 | DMG / Mac App Store / Homebrew |
| Linux | PyInstaller | FHS 目录规范 | AppImage / deb/rpm / Snap/Flatpak |

### 3.3 自动更新机制

- 增量更新 + 三通道（稳定版/测试版/开发版）
- 强制更新机制 + 一键回滚

### 5.1 全链路合规与法律风险管控

## 核心避坑指南

1. 不要一开始就追求大而全 —— 先把核心的幻觉核查能力做到极致
2. 不要把所有希望寄托在单一外部依赖上 —— 必须做适配层隔离
3. 不要忽视合规问题 —— 知识产权、隐私保护、监管合规是底线
4. 不要闭源核心代码同时又不做保护 —— 代码混淆、加壳保护
5. 不要脱离用户迭代 —— 所有功能基于真实反馈
