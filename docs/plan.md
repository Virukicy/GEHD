## 六、产品定位 —— 人机双轨战略

> **GEHD 的引擎给 AI 用，GEHD 的产品给人用。**

### 6.1 为什么这个定位？

GEHD 诞生于「AI 代理需要规则引擎核查文档」的场景。但纯规则引擎给 AI 代理用的价值很薄——任何 AI 代理被赋予「用正则扫描可疑专有名词」的指令都能写出类似逻辑。

GEHD 真正的不可替代性在于：**把规则引擎 + 联网核查 + LLM 过滤 + 审计追溯组合成无需编程的完整闭环。** 这个闭环的核心消费者是需要核查文档真实性、但不想写代码的人。

### 6.2 双轨定义

| 轨道 | 面向 | 交互方式 | 优先级 | 迭代重点 |
|------|------|------|:--:|------|
| **产品轨** | 人类用户 | GUI 桌面应用 | 🟠 主 | 管道状态栏、全文高亮、一键核查、审计视图、跨平台分发 |
| **引擎轨** | AI 代理 | CLI / API | 🔵 辅 | 保持接口稳定、输出格式清晰、零额外维护成本 |

### 6.3 决策准则

- 新功能优先考虑「人需要什么」而非「AI 代理能用什么」
- AI 代理轨道不需要额外优化体验——CLI 输出结构化 JSON 即可
- GUI 是产品轨的核心投入方向，AI Guide 从主文档降级为参考附录
- 引擎层是两条轨道的唯一共享资产，所有变更必须同时兼容 CLI 和 GUI

## 七、v0.5.0 展望 —— 可审计性基础设施

> v0.4.0 管道架构预留 `PipelineContext.decision_log: list[dict]` trace 字段，v0.5.0 实现完整决策追溯链。

### 7.1 为什么可审计性先于一切

v0.4.0 引入 LLM 管道后，GEHD 面临一个根本性矛盾：

- **规则引擎时代**：每个决策可追溯到具体正则匹配和评分公式 → 天然可审计
- **LLM 管道时代**：前置过滤和后置纠正都是黑盒 → **审计能力断崖式下降**

v0.5.0 的核心使命是**重新闭合审计链路**——让 LLM 管道的每一步决策都能被追溯。

### 7.2 交付范围

| 项 | 说明 |
|------|------|
| `decision_log` 完整填充 | 每个管道阶段向 context.decision_log 写入结构化 trace 条目（触发原因、输入、输出、置信度） |
| 审计视图（GUI） | 新建一个「审计」面板：选中任意候选实体 → 展开完整决策链 → 支持折叠/跳转/导出 |
| 评分可解释性增强 | 每个候选项的评分明细从隐式（代码内）变为显式（decision_log 中列出各维度加减分） |
| CLI 审计模式 | `--audit` 标志输出机器可读的完整 decision_log JSON，供外部审计工具消费 |

### 7.3 契约式管道架构

v0.4.0-rc 暴露了信任式管道的根本缺陷：各阶段假设上游数据格式正确，不作验证，接线故障静默跳过。v0.5.0 升级为契约式管道。

**核心改动**：每个管道阶段声明 `requires`（输入必须非空）和 `produces`（输出字段名），管道编排器运行时验证契约。

```python
STAGE_CONTRACTS = {
    'llm_pre':  {'requires': ['candidates'],       'produces': ['candidates', 'decision_log']},
    'llm_post': {'requires': ['l4_queue'],          'produces': ['decision_log']},
    'web_verify': {'requires': ['l4_queue'],        'produces': ['l4_queue']},
}
```

**收益**：

| 对比 | 信任式（v0.4.0） | 契约式（v0.5.0） |
|------|------|------|
| 输入验证 | `context.get('field', [])` → 空就空 | 声明 `requires` → 缺失写 decision_log 并跳过 |
| 失败模式 | 静默跳过，无人知晓 | `decision_log` 显式记录「阶段 X 因缺字段 Y 跳过」 |
| 阶段测试 | 必须跑全管道 | fixture 数据即可独立测单阶段 |
| 接线排查 | 逐行 trace 代码 | 看 decision_log 即可定位 |
| 新阶段接入 | 手动检查上下游字段 | 加一条 `STAGE_CONTRACTS` 条目，自动验证 |

**改动量**：`engine/pipeline.py` +30 行（新增契约定义 + 验证函数），各阶段文件**零改动**。

### 7.4 审计视图示例

用户在 GUI 中点击一个标记为「中危」的实体，审计面板展开：

```
📋 实体: "辰星微电子"  |  最终判定: 未通过（中危 60分）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  [L3 规则评分]
  正则匹配: 半导体企业名 (基础分 45)
  形容词前缀: 无
  电商平台加分: +0
```

## 八、v0.6.0 展望 —— 自迭代闭环

> 引擎越用越聪明，用户越用越省心。

### 8.1 核心逻辑

v0.5.0 建立了审计安全网之后，可以安全地让引擎自动调整自身配置。

### 8.2 两档模式

| 模式 | 行为 | 适用用户 |
|------|------|------|
| **建议模式**（默认） | 引擎扫描 → 联网验证 → 自动判断应加白/黑名单 → **弹出建议，用户确认后生效** | 所有用户 |
| **自动模式**（可选开启） | 同上，但无需确认，直接写入配置 | 信任引擎准确率的进阶用户 |

### 8.3 交付范围

| 项 | 说明 |
|------|------|
| 配置建议引擎 | 基于 L4 联网核查结果自动判断：已验证真实 → 建议加白名单；已验证虚构 → 建议加黑名单；频繁误报 → 建议加排除词 |
| 建议审核 UI | GUI 设置页新增「待审核建议」面板：展示引擎建议的配置变更 → 逐条审批或批量通过 |
| 自动模式开关 | 设置页新增「自动学习」开关（默认关闭） |
| 审计保护 | 所有自动变更写入 decision_log，可在审计视图回溯 |

### 8.4 风险控制

- 自动模式仅影响白名单/黑名单/排除词三类配置，不动评分阈值和正则规则
- 每次自动变更前比对 `_l4_cache.json` 历史记录，避免反复横跳
- 所有变更可撤销（GUI 中一键回滚到上次手动确认的快照）

## 九、v0.7.0 展望 —— 跨平台分发 + 领域拓展

### 9.1 时机判断

v0.7.0 之前，GEHD 经历了：
- v0.4.0：管道完善 → 核心能力稳定
- v0.5.0：审计完善 → 可信赖
- v0.6.0：自进化 → 维护成本下降

此时核心产品已经「好用」且「可信」。v0.7.0 是**放大的时刻**——让更多人能用，让更多场景能覆盖。

### 9.2 跨平台分发

| 平台 | 打包方案 | 分发形式 |
|------|------|------|
| macOS | PyInstaller (.app) + create-dmg | DMG 安装包 |
| Windows | PyInstaller + Inno Setup | .exe 安装包 |
| Linux | PyInstaller | AppImage |

- 三通道自动更新：stable / beta / dev
- 更新检测在 GUI 启动时静默检查，弹出更新提示

### 9.3 领域拓展（管道 + 适配层的自然延伸）

v0.4.0 建立的管道架构天然支持领域插拔。v0.7.0 通过新增配置和规则文件，扩展核查领域：

| 新领域 | 可能需要的新能力 | 复用已有 |
|------|------|------|
| 合同核查 | 金额矛盾检测增强、条款编号一致性 | L2.5 统计数据、L3.6 一致性 |
| 财报核查 | 数字交叉验证、增长率合理性 | L2.5 统计数据、L4 联网核查 |
| 新闻核查 | 引述追溯、时间线校验 | L2.5 引述、L2.5 时间线 |
| 学术论文 | 引用真实性、数据来源追溯 | L3.7 声明提取、L4 联网核查 |

实现方式：每种新领域 = 一套额外的 `config/{domain}_patterns.json` + L1-L3.7 现有逻辑复用，**无需改核心代码**。
