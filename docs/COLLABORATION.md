## 三、条条铁律

1. **开工前 `git pull`** — 确保拿到最新代码
2. **只改自己域内文件** — 碰共享/冻结文件前先喊话
3. **改完立即 `git push`** — 不让其他组等
4. **QA 有哨卡通过权** — 未通过哨卡的 commit，对应团队需修复

## 四、标准工作流

### E / U 开工

```bash
git pull
# 改代码（只动自己域内的文件）
git add -A
git commit -m "Team: 做了什么"
git pull --rebase
git push
```

### D 开工

D 组修改文档后同样遵循标准 Git 流程，确保其他组能拿到最新文档：

```bash
git pull
# 修改 docs/ 或 README.md
git add -A
git commit -m "D: 做了什么"
git pull --rebase
git push
```

**commit message 格式**：`D: {PM2D 传递号} {一句话描述}`
例如：`D: PM2D-009 COLLABORATION.md 传递规范升级`

**D 组文档变更规则**：
- 仅修改 `docs/` 和 `README.md`，不动 `src/`
- 每次 PM 指令执行完毕后立即 `git push`
- 如遇冲突（他人同时改了同一文档），按 §五 冲突处理规则执行

### 提交后 QA 哨卡

QA 每次提交后自动执行：

```
pytest                    ← 全量测试是否通过？
mypy src/                ← 有无新增类型错误？
ruff check src/          ← 有无新增 lint 问题？
git diff --stat          ← 是否踩到其他组的文件域？
```

**通过** → QA 产出简短确认。**不通过** → QA 产出 `Q2E-YYYYMMDD-NNN`（E）或 `Q2U-YYYYMMDD-NNN`（U），列明问题。

### `--rebase` 说明

如果你的 commit 在对方之后才推，`rebase` 会把你的改动接在对方后面，避免分叉的 merge commit。

```
对方: A → B → C
你:   A → B → D        ← 先 rebase，变成 A → B → C → D，再 push
```

## 五、冲突处理

| 场景 | 怎么做 |
|------|------|
| 同时改**不同文件** | `git pull --rebase` 自动合并 |
| 同时改**同一文件** | 文件域分治下不应发生。如发生，后 push 的人手动解决 |
| 需改**冻结共享文件** | 对话喊话相关方，同意后再改 |
| 需改**对方域文件** | 对话提需求，让对方改 |
| QA 审查不通过 | 修复或通过传递文档回复，不让审查项悬空 |

## 七、PM 指令与调度协议

### 7.1 PM 指令类型

PM 传递文档通过措辞区分指令类型。各组收到后须按对应流程执行。

**A 类：直接执行**

措辞特征：「执行」「完成」「修复」「更新」「迁移」「提交」。无前置条件。

流程：收到 → 执行 → 回执。

**B 类：先审后做**

措辞特征：「设计草案」「方案」「PM 确认后执行」「先出方案」「审批」。

流程：收到 → 产出草案/方案 → 回执附上 → 等待 PM 审批 → PM 批了再动工。

B 类指令的典型场景：新功能设计、架构变更、跨组影响的改动。

**C 类：信息通知**

措辞特征：「通知」「了解即可」「无需操作」「知会」。

流程：收到 → 阅读 → 无需回执。

### 7.2 PM 调度环

PM 通过以下循环分步推进跨组工作：

> PM 决策 → 助理出方案 → PM 审批 → 分步下发各组 → 各组回执 → 助理汇总 → PM 审阅 → 下一轮

操作原则：
- 有依赖关系的指令按顺序下发（如 E 建目录 → D 改协议 → 其他人迁移）
- 无依赖关系的指令并行下发（如 U 和 QA 同时迁移各自文件）
- 各组只写自己文件域，不同时操作同一文件则零冲突
- 全部回执后 PM 发 PM2A 宣布闭环

### 7.3 各组回执义务

| 指令类型 | 必须回执 | 回执内容 |
|------|:--:|------|
| A（直接执行） | 是 | 执行结果 + commit hash |
| B（先审后做） | 是 | 草案/方案，等审批后再执行 |
| C（信息通知） | 否 | — |

回执使用对应传递号前缀（E→PM 用 `E2PM`、U→PM 用 `U2PM`、QA→PM 用 `Q2PM`、D→PM 用 `D2PM`）。

### 7.4 决策文档化

PM 的重大决策（团队调整、流程变更、架构裁定）应在 `workspace/PM/decisions/` 下存档，文件名格式 `YYYY-MM-DD_简述.md`，方便后续追溯。

### 7.5 功能闭环循环

每项引擎功能完成（如 P2-3）后，在进入下一项功能（如 P2-4）之前，执行以下全组闭环：

| 顺序 | 谁 | 做什么 |
|:--:|:--:|------|
| 1 | QA | 对新增代码跑哨卡审查 |
| 2 | D | 同步架构文档、CHANGELOG |
| 3 | U | 自查新功能是否触发 GUI 变更需求 |
| 4 | PM | 汇总三方回执 → 决策下一阶段 |

第 1-3 步并行下发，各组只写自己文件域，无冲突。全部回执后 PM 批准 E 进入下一功能。

### 7.6 执行证据标准

当任何团队的回执声称"已运行工具/命令"时，回执内容必须包含可验证的执行产物。LLM 推理生成的内容不被视为执行证据。

**A. 原始输出强制附带**

凡回执中引用了命令执行结果（如 GEHD 扫描输出、pytest 结果、mypy 报告），必须逐字附带该命令的完整终端输出，不得删减、归纳或"分析后重述"。

**B. 产物引用**

命令执行生成的文件（如 `_l4_queue.json`、`coverage.json`）必须报告：
- 文件完整路径
- 关键数字（如实体数、测试数），不可仅描述"内容"
- 这些数字须与终端输出一致

**C. 段落号可追溯**

凡引用扫描结果（issues、warnings），每条须附带段落号，与终端原始输出中的段落号严格一致。

**D. PM 保留独立验证权**

PM 可自行运行相同命令交叉验证任何团队回执中的数字。若数字不一致，视为回执无效，发回重做。

### 7.7 S 组赋权说明

S（战略规划）组拥有**建议权**，PM 拥有**决策权和调度权**。

- S 可向 PM 提交路线图、架构方案、技术分析报告
- PM 决策后将形成 PM2X 指令，分发至各执行组
- S 不直接对 E/U/QA/D 发送指令或跨组协调
- S 与 PM 的通信使用 `S2PM` / `PM2S` 前缀，S 向全体广播使用 `S2A`

> S 组文档传输文件夹：`workspace/S/transfer/`。PM 可从此文件夹拉取 S 产出的长文档。

### 7.8 接口冻结纪律

任何跨 E/U 的功能，强制执行「先冻结，后并行」五步法：

| 步骤 | 谁 | 做什么 | 冻结物 |
|:--:|:--:|------|------|
| 1 | E | 完成接口定义，写入 `workspace/E/interface-{version}.md` | 字段名、类型、示例值 |
| 2 | S | 审阅接口，确认与 plan.md 一致 | — |
| 3 | PM | PM→U 下发冻结后的接口文档 | — |
| 4 | E+U | 并行开发，E 不改接口，U 只读冻结版本 | 冻结物不可变 |
| 5 | QA | 按接口文档验证 E/U 产出对齐 | — |

**铁律**：
- 步骤 3 完成前，U 不动代码
- 步骤 3 完成后，E 不改接口
- 任何接口变更 → 重新走步骤 1-3

**来源**：v0.5.0 架构革新教训——E 和 U 同时改动、接口未冻结、三模式同时暴雷后反复排查。

### 7.9 过程哲学 — 模型分工与增量交付

GEHD 团队使用分层模型策略，以模型能力差异强制执行工程纪律：

| 组 | 模型 | 原因 | 交付纪律 |
|:--:|:--:|------|------|
| E / U / QA / D | DeepSeek Flash | 低成本，天然限制一次交付规模 | 一次 ≤3 个源文件，一个模块一个 commit |
| PM / S | DeepSeek Pro | 战略决策与架构推理需全量上下文 | PM 掌控审批/调度/闭合；S 产出分析报告 |

**设计哲学**：Flash 的能力天花板 = 天然强制执行增量交付。v0.5.0 架构革新过于激进（E/U 同时动、三模式同时暴雷），正是因为 Pro 模型能"一次写五个文件"的便利性消解了谨慎。Flash 强制"一次一个模块"的节奏，从模型层面杜绝大爆炸式变更。

**不是省钱策略，是工程纪律策略。**

### 7.10 身份卡 — 智能体上下文注入

GEHD 执行智能体（E/U/QA/D）使用 Flash 模型，每次会话独立。**PM 每次调度必须携带身份卡**。

#### 强制注入规则

- PM 向任何执行组发 A 类指令时，消息体必须含身份卡
- 身份卡位于 `## 背景` 之前，作为独立区块
- 按收件组定制，不得用通用模板

#### 身份卡模板（以 PM2E 为例）

```
## 🪪 身份卡

- **你是**：E · 引擎工程师
- **项目**：GEHD 实体幻觉检测工具 v0.5.2
- **你的文件域**：`engine/` `io/` `cli/` `config/` `tests/`（引擎测试，不含 `test_gui.py`）`CHANGELOG.md` `pyproject.toml`

  - `tests/conftest.py` 改动前须通知其他组
- **你的产出格式**：E2PM 回执，含变更文件路径 + commit hash + 验收命令输出
- **绝对禁止**：
  - 禁止扮演 PM、S、U、QA、D
  - 禁止起草 PM2A 或任何 A 类传递号
```

## 八-A、PM 短指令词典

PM 对人机对话使用简练短指令调度智能体。以下为各智能体必须识别的指令及其含义。

### 对执行组（E / U / QA / D）

所有执行组理解以下指令模式（将 `{prefix}` 替换为对应前缀）：

| 指令 | 含义 | 执行动作 | 示例 |
|------|------|------|------|
| `{prefix}` | 查最新一份 PM2{角色} | 读取 `.workbuddy/` 下当天日期最新序号的 PM2{角色} 文件 | 对 D 说 `PM2D` → 读当天最新的 PM2D-YYYYMMDD-NNN.md |
| `{prefix} NNN` | 查指定编号 | 读取 `.workbuddy/` 下当天该 NNN 序号的 PM2{角色} 文件 | 对 D 说 `PM2D 005` → 读 PM2D-20260517-005.md |
| `{prefix} N` | 查最近 N 份 | 读取当天最近 N 份 PM2{角色} 文件（按序号降序取 N 个） | 对 D 说 `PM2D 3`，最新为 008 → 读 006、007、008 |
| `PM指令` | 查 PM 最新指令 | 同 `{prefix}`，读取当天最新一份发给自己的 PM2{角色} 文件 | 对 E 说 `PM指令` → 读当天最新的 PM2E-*.md |

**前缀对应表**：

| 角色 | {prefix} | 查找文件模式 |
|:--:|------|------|
| E | `PM2E` | `.workbuddy/PM2E-*` |
| U | `PM2U` | `.workbuddy/PM2U-*` |
| QA | `PM2QA` | `.workbuddy/PM2QA-*` |
| D | `PM2D` | `.workbuddy/PM2D-*` |

**查找规则**：
- 如指令未指定日期，默认查当天（`YYYYMMDD` = 当前日期）
- `{prefix} N` 模式按序号降序排列，取前 N 个
- 无匹配文件时回复「当日无 PM2{角色} 指令」

### 对 PM

PM 智能体识别以下回执查阅指令：

| 指令 | 含义 | 执行动作 |
|------|------|------|
| `{role}2PM` | 查 {role} 最新回执 | 读取 `.workbuddy/` 下当天最新序号的 {role}2PM 文件 |
| `{role}回执` | 同上 | 同 `{role}2PM`，语义相同 |
| `{role}2PM NNN` | 查指定编号回执 | 读取当天该 NNN 序号的 {role}2PM 文件 |
| `{role}2PM N` | 查最近 N 份回执 | 读取当天最近 N 份 {role}2PM 文件 |

**回执前缀对应表**：

| 指令 | 查找文件模式 |
|------|------|
| `D2PM` / `D回执` | `.workbuddy/D2PM-*` |
| `E2PM` / `E回执` | `.workbuddy/E2PM-*` |
| `U2PM` / `U回执` | `.workbuddy/U2PM-*` |
| `Q2PM` / `QA回执` | `.workbuddy/Q2PM-*` |

> ⚠️ 此词典为智能体必须遵循的行为规范。PM 使用短指令时，智能体不得回复"请说明需要什么"或"请提供更多上下文"——直接查找并执行。
